# Schedula FastAPI — Development Instructions

## Architecture Principles

**Core Rule:** FastAPI is stateless and write-free. Next.js owns all persistence.

- FastAPI reads from MongoDB, runs OR-Tools solver, returns snapshot (never writes schedule to DB)
- Only Next.js writes approved schedules to `schedules` collection
- Every MongoDB query must filter by `institution_id` — no cross-institution data leakage
- Uvicorn runs on Azure Container Apps and scales to zero

## FastAPI Conventions

### 1. Async Everything

- Use `async def` for all route handlers and database queries
- Use PyMongo Async (`pymongo.asynchronous`) for async MongoDB operations
- Never use `time.sleep()` — use `asyncio.sleep()` instead

### 2. Dependency Injection

```python
async def get_db():
    # Return PyMongo Async database connection
    return db

@app.get("/endpoint")
async def handler(db: AsyncDatabase = Depends(get_db)):
    pass
```

### 3. Request/Response Models (Pydantic)

- All request bodies must have a Pydantic model in `app/models/`
- Use `field_validator` for custom validation
- Use `ConfigDict(populate_by_name=True)` for flexible field naming
- Response models should be clear about what's returned

Example:

```python
from pydantic import BaseModel, field_validator

class CourseSection(BaseModel):
    section_id: str
    type: Literal["lecture", "lab", "tutorial"]
    year_levels: list[int]
    slots_per_week: int
    capacity: int
    required_room_label: str
    shared_with: list[str] = []  # Department IDs

    model_config = ConfigDict(populate_by_name=True)

    @field_validator("year_levels")
    @classmethod
    def validate_year_levels(cls, v):
        if not v or any(y < 1 for y in v):
            raise ValueError("year_levels must be non-empty positive integers")
        return v
```

### 4. Error Handling

- Use FastAPI's `HTTPException` for API errors
- Include descriptive error messages and appropriate status codes
- Log errors with context (institution_id, user_id if applicable)

```python
from fastapi import HTTPException

if not institution:
    raise HTTPException(status_code=404, detail="Institution not found")
```

### 5. Environment & Configuration

- All config in `app/config.py` using Pydantic `Settings`
- Read from `.env` file during development
- Never hardcode secrets or MongoDB URIs

## MongoDB Async Patterns (PyMongo Async)

### Async Database Queries

```python
# All network operations are coroutines
institution = await db["institutions"].find_one({"_id": institution_id})

# Bulk operations
result = await db["course_sections"].insert_many(sections)

# Aggregation pipeline
pipeline = [
    {"$match": {"institution_id": institution_id}},
    {"$group": {"_id": "$department_id", "count": {"$sum": 1}}}
]
stats = await db["course_sections"].aggregate(pipeline).to_list(None)
```

### Filters & Projections

- Always filter by `institution_id` on every query
- Use projections to minimize data transfer
- Never assume data exists — handle `None` cases

```python
async def get_courses(db, institution_id: str):
    courses = await db["course_sections"].find(
        {"institution_id": institution_id},
        {"name": 1, "capacity": 1, "required_room_label": 1}
    ).to_list(None)
    return courses
```

### Distinct Values

```python
departments = await db["course_sections"].distinct(
    "department_id",
    {"institution_id": institution_id}
)
```

## Solver Integration (OR-Tools)

### Key Rule: Stateless

Solver runs in-memory. Input all data in one call, return schedule snapshot.

### Phase 3 Workflow

1. **Load Data**: Read institution, staff, courses, rooms, availability from MongoDB
2. **Build Model**: Create OR-Tools CpModel with all constraints
3. **Solve**: Run solver.Solve(model, strategy) — captures solution
4. **Format Response**: Convert solver results to schedule JSON
5. **Return**: Send snapshot to Next.js (no DB write)

**Pattern Reference**: Google's [Employee Scheduling](https://developers.google.com/optimization/scheduling/employee_scheduling) tutorial covers the same CpModel patterns — variables (shifts), constraints (no overlap, capacity), and objectives. Highly applicable to Schedula's course scheduling.

### Constraint Implementation

- **H1–H5, H7**: Standard scheduling constraints (capacity, no overlap, day-off)
- **H6**: Time window constraints (institution daily hours)
- **H8**: Year-level conflict detection per department
- **H9**: Cross-department shared lecture conflict propagation

### Soft Constraint Weights

- S1 (100): Break window violations
- S2 (80): Consecutive slots penalty per staff
- S3 (60): Session spread penalty
- S4 (40): Campus day clustering penalty

Coordinator can adjust weights before generation. Default to above.

Example structure:

```python
from ortools.sat.python import cp_model

model = cp_model.CpModel()

# Add variables
for section_id, time_slot in combinations(sections, time_slots):
    assigned[(section_id, time_slot)] = model.NewBoolVar(f"assign_{section_id}_{time_slot}")

# Add hard constraints
for room_id in rooms:
    for time_slot in time_slots:
        model.AddAtMostOne([
            assigned[(s, t)] for s in sections
            if sections[s]["room_id"] == room_id and t == time_slot
        ])

# Add soft constraints (penalties)
for staff_id in staff:
    consecutive_penalty = sum([
        (assigned[(s, t)] * soft_weights["S2"])
        for s, t in constraints_consecutive(staff_id)
    ])
    model.Minimize(consecutive_penalty)

solver = cp_model.CpSolver()
status = solver.Solve(model)
```

## API Endpoint Standards

### Naming & Paths

- Lowercase, plural nouns: `/institutions`, `/courses`, `/availability`
- Specific actions: `POST /schedule/generate`, `GET /schedules/{id}`
- Version prefix optional: `/v1/...` or just `/...`

### Response Format

All responses follow:

```python
{
    "data": {...},
    "meta": {
        "count": int,
        "institution_id": str,
        "timestamp": datetime
    },
    "error": null  # or error details
}
```

### Common Endpoints

```
POST   /generate-schedule          → trigger solver, return snapshot
GET    /institutions/{id}          → institution profile
POST   /courses/import             → bulk import sections
POST   /availability/submit        → staff submits availability
GET    /schedule/{id}              → retrieve snapshot (read-only)
```

## Key Invariants & Rules

1. **No Cross-Institution Leakage**: Every query filters by `institution_id`
2. **Schedule Never Persists on FastAPI**: Only snapshots returned; Next.js owns the DB write
3. **Stateless Solver**: Input all data once, return complete snapshot
4. **Day-Off is Hard Constraint**: Cannot be overridden by soft weights
5. **Unsubmitted Staff = Fully Available**: No availability data = treat as always free
6. **Labs & Tutorials Never Shared**: Only lectures can have `shared_with` property
7. **One Approved Schedule Per (institution_id, term_label)**: Unique index enforced by Next.js (MongoDB)
